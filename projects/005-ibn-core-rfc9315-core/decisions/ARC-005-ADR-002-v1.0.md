## 3. Context and Problem Statement

### 3.1 Problem Description

ADR-001 decided to extract the RFC 9315 phase machine into a layer-agnostic core that both the business-intent agent (BSS) and resource-intent agent instantiate as peer `PhaseStrategy` adapters. That extraction is now **substantially complete in code**, ahead of the roadmap calendar:

- **Phase 1 (Extract Core)** — DONE: `src/imf/core/{IntentCycleRunner,PhaseStrategy}.ts` on `main`; `IntentHandlingCycleRunner` slimmed to delegate to the core via `BssPhaseStrategy` (public API unchanged).
- **Phase 2 (Package)** — DONE: git-installable library entry; entry slimmed so consumers no longer drag `@anthropic-ai/sdk` (closes D4) and the phase enums are exported (closes D6) — via PRs #51/#54/#55.
- **Phase 3 (Resource Adopts)** — DONE: resource-intent-agent consumes the core (Project 004), pinned to the SDK-free `6350d96` (interim, per ADR-009).

The roadmap (`ARC-005-ROAD`) names the remaining gates: **Gate B (TMF921 CTK-parity)** → **Gate C (v3.0.0 release)**. The question is whether to cut `v3.0.0` now.

**Problem statement as a question**: Should we cut the `v3.0.0` release tag now that the cutover code is merged, or reserve the number and defer the tag until the conformance gate is satisfiable?

### 3.2 Why This Decision Is Needed

- **Two constraints collide**: (a) PRIN 3 (Standards Conformance, NON-NEGOTIABLE) requires TMF921 **CTK 83/83 recorded** + the canonical O2C green for any change to the conformance-bearing BSS path — and the BSS-on-core delegation *is* such a change; (b) CLAUDE.md mandates that release tags are **immutable and cited in publications** — a tag cannot be walked back.
- **The gate is currently unsatisfiable in the working environment**: the formal CTK needs the external CTK harness, and the live O2C is blocked on exhausted Anthropic API credits. Only *local proxy* evidence exists (Gate B green on locally-executable checks; formal CTK never run).
- **A version-number conflict must be resolved**: CLAUDE.md's tag table lists `v3.0.0 = "Live CAMARA integration"` (authored before Project 005), while `ARC-005-ROAD` claims `v3.0.0` for the core cutover.

## 4. Decision Drivers (Forces)

### 4.1 Technical Drivers

- **Conformance integrity**: cutting a release that changes the BSS runner path without recorded CTK 83/83 would assert conformance we have not verified.
- **Tag immutability**: a mis-cut `v3.0.0` cannot be corrected without violating the never-rewrite-tags rule.

### 4.2 Business Drivers

- **Credibility**: ibn-core's commercial + academic value rests on verifiable conformance (PRIN 3). A premature `v3.0.0` undermines it.
- **Consumer stability**: the resource agent's pin should advance to a *stable tag* only once that tag is sound.

### 4.3 Regulatory & Compliance Drivers

- PRIN 3 (Standards Conformance, NON-NEGOTIABLE); PRIN 16/17 (CTK not regressed; traceability).

### 4.4 Alignment to Architecture Principles

| Principle | Alignment | Impact |
|-----------|-----------|--------|
| PRIN 3 — Standards Conformance | ✅ Supports | Gate the release on recorded CTK 83/83 |
| PRIN 14 — Maintainability | ✅ Supports | One conformance-tested phase machine; clear release discipline |
| PRIN 9 — Open-Core Seam | ✅ Supports | Cutover is the clean port-set seam reaching GA |

## 5. Considered Options

### Option 1: Reserve v3.0.0, defer the tag until Gate B is green — CHOSEN

**Description**: Record the cutover-readiness (Phases 1–3 done), **reserve** `v3.0.0` in CLAUDE.md for the core cutover (resolving the conflict by moving CAMARA/federation to later numbers), and document the two deferred actions — cut `v3.0.0` + re-pin resource ADR-009 `6350d96` → `v3.0.0` — as gated on Gate B (CTK 83/83 + live O2C).
**Wardley stage**: governance/process (n/a).

- ✅ Honours PRIN 3 and the immutable-tag rule.
- ✅ Resolves the version-number conflict now (cheap, reversible).
- ✅ Keeps the consumer on the sound interim pin (`6350d96`).
- ❌ The "two peers, one core" milestone isn't formally released yet (a comms/marketing delay only).

### Option 2: Cut v3.0.0 now (waive the gate)

- ✅ Formal milestone immediately; resource re-pins to a clean tag.
- ❌ Asserts unverified TMF921 conformance — **breaches PRIN 3 (NON-NEGOTIABLE)**.
- ❌ Immutable tag on unproven conformance cannot be corrected — reputational/academic risk.
- **Rejected.**

### Option 3: Keep CLAUDE.md as-is (v3.0.0 = CAMARA), use a different number for the cutover

- ✅ No edit to the legacy tag table.
- ❌ Contradicts `ARC-005-ROAD` (which gates v2.x → v3.0.0 on the cutover) and the established "breaking change = v3.0.0" framing; fragments the semver story.
- **Rejected** — the roadmap (newer, ARB-approved) supersedes the pre-005 table line.

### Option 4: Do Nothing (Baseline)

- ❌ Leaves the version conflict unresolved and the cutover-readiness unrecorded; future agents re-litigate. **Rejected.**

## 6. Decision Outcome

### 6.1 Chosen Option

**"Option 1 — reserve v3.0.0, defer the tag until Gate B is green."**

### 6.2 Y-Statement

> **In the context of** the RFC 9315 core cutover whose code (BSS + resource on one core) is already merged,
> **facing** an unverifiable TMF921 CTK-parity gate in this environment and the immutability of release tags,
> **we decided for** reserving `v3.0.0` for the cutover and deferring the tag + resource re-pin until CTK 83/83 + live O2C are recorded green,
> **to achieve** a release that honours PRIN 3 and the never-rewrite-tags rule,
> **accepting** that the "two peers, one core" milestone is announced later than the code landed.

### 6.3 Justification

The cutover is **done in substance** (Phases 1–3 merged); the only outstanding items are a *gated* release tag and a consumer re-pin. Cutting the tag now buys a marketing milestone at the cost of breaching a NON-NEGOTIABLE principle on an immutable artifact — a bad trade. Reserving the number resolves the conflict cheaply and reversibly and keeps the consumer on a sound pin.

**Risk appetite**: conformance-credibility appetite is effectively zero; deferring the tag is the only option within it.

## 10. Implementation Plan (deferred runbook — execute when Gate B is green)
