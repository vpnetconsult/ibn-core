## 1. Decision Title

**Upgrade the OpenTelemetry suite in the business-intent agent across a breaking boundary — `@opentelemetry/sdk-node` `^0.218.0`→`^0.219.0`, `@opentelemetry/exporter-trace-otlp-http` `^0.218.0`→`^0.219.0`, `@opentelemetry/auto-instrumentations-node` `^0.75.0`→`^0.77.0`, `@opentelemetry/resources` `^2.7.0`→`^2.8.0` (pulling `@opentelemetry/core`, `resources`, `sdk-trace-base` to 2.8.0) — to remediate GHSA-8988-4f7v-96qf (unbounded memory allocation in W3C Baggage propagation).**

## 3. Context and Problem Statement

A production `npm audit` of the business-intent agent (`src/`) reported **26 moderate** advisories, all in the OpenTelemetry suite pinned at `0.218.0` / core `2.7.1`. The root advisory is **GHSA-8988-4f7v-96qf — "OpenTelemetry Core: unbounded memory allocation in W3C Baggage propagation"**, fixed in `@opentelemetry/core` `>= 2.8.0`.

This is **reachable, not theoretical**: `src/telemetry.ts` installs `getNodeAutoInstrumentations()`, which parses incoming HTTP headers — including the W3C `baggage` header — on every request. A crafted `baggage` header can therefore drive unbounded allocation in the deployed agent (a moderate-severity DoS).

The fix crosses a **breaking** semver boundary (`sdk-node` `0.x` minor; core `2.7`→`2.8`), and it touches a **named architectural seam**: per `CLAUDE.md`, `src/telemetry.ts` MUST remain the first import in `src/index.ts` (the OTel SDK hooks `require` at init; anything imported earlier never emits spans). That combination — breaking change + named seam — is exactly what the ArcKit flow requires an ADR for, rather than a drive-by lockfile bump.

**Problem statement**: How do we remediate the reachable baggage DoS without regressing the UC006 telemetry seam (span emission, GenAI/`rfc9315.phase` attributes) or the canonical O2C?

**Requirements/links**: NFR security hygiene for the deployed image; preserves UC006 (Custom Observability) and the Standards Map entry for `src/telemetry.ts`. Companion to the non-breaking `fix(deps)` PR #65 (hono/form-data) which deliberately deferred this breaking item.

---

## 4. Decision Drivers

- **Reachable DoS** — the baggage advisory is exercised by the always-on auto-instrumentation HTTP path.
- **Clean audit floor** — take the deployed image's production audit to **0** vulnerabilities.
- **Seam integrity (UC006)** — `telemetry.ts` must keep emitting spans; the bootstrap-first-import rule is non-negotiable.
- **No behaviour change to the RFC 9315 cycle** — telemetry is cross-cutting; the intent cycle and O2C contract must be unchanged.
- **Proportionate blast radius** — bump the meta-packages, let the resolver pull the fixed transitives, avoid hand-pinned overrides.

---

## 5. Considered Options

### Option 1: Bump the OTel meta-packages across the breaking boundary — CHOSEN

Raise the four direct OTel deps to `sdk-node ^0.219.0` / `exporter-trace-otlp-http ^0.219.0` / `auto-instrumentations-node ^0.77.0` / `resources ^2.8.0`; the resolver pulls `core`/`resources`/`sdk-trace-base` → `2.8.0`. Lockfile-regenerated; `package.json` floors raised so the fix cannot regress.

- ✅ Closes GHSA-8988-4f7v-96qf at source; production audit 26 → **0**.
- ✅ `telemetry.ts` API usage (`NodeSDK`, `OTLPTraceExporter`, `getNodeAutoInstrumentations`, `resourceFromAttributes`) is unchanged across the bump — no code edit needed.
- ❌ Breaking semver boundary — requires a build (tsc) + in-cluster telemetry smoke test to confirm no runtime regression.

### Option 2: `overrides` to force `@opentelemetry/core@2.8.0` only

Pin the transitive `core` via npm `overrides`, leave `sdk-node`/exporters at 0.218.

- ✅ Smaller version delta on paper.
- ❌ Fights the resolver (sdk-node 0.218 expects core 2.7.x); risks API mismatch and silent breakage of the SDK; overrides are opaque and easy to lose. Higher real risk than Option 1.

### Option 3: Do Nothing (baseline)

- ✅ No change.
- ❌ Ships a reachable moderate DoS in the deployed image; production audit stays non-zero; defers an unavoidable upgrade while the gap widens.

---

## 6. Decision Outcome

**Chosen: Option 1.**

> In the context of a reachable W3C-baggage DoS in the deployed business-intent agent,
> facing a fix that crosses a breaking OTel boundary and touches the `telemetry.ts` seam,
> we decided to bump the OTel meta-packages to `sdk-node 0.219` / `core 2.8` and regenerate the lockfile,
> to take the production audit to zero without changing the RFC 9315 cycle or the UC006 span contract,
> accepting a breaking-version delta that we validate with an image build (tsc) plus an in-cluster telemetry + O2C smoke test.

---

## 7. Consequences

### Positive

- Production `npm audit` for the agent image: **0** vulnerabilities (was 26 moderate).
- Baggage DoS closed at the always-on instrumentation path.
- OTel floors raised in `package.json` — cannot silently regress below the fix.

### Negative (accepted)

- Breaking-version bump; relies on the build + smoke test (below) rather than CI to catch regressions (Actions is billing-locked).
- `auto-instrumentations-node 0.77` may add/adjust default instrumentations vs 0.75 — span shape for non-core libraries could shift slightly (telemetry-only, non-functional).

### Neutral

- Dev-only jest/babel advisories remain (do not ship); out of scope.
- A separate, later upgrade may move OTel again; this ADR sets the current floor, not a freeze.

---

## 8. Validation

- **Build/compile**: `docker build` of `src/` (runs `npm ci` against the regenerated lockfile + `tsc`) must succeed — proves API compatibility of `telemetry.ts` with core 2.8 / sdk-node 0.219.
- **Telemetry seam**: after redeploy, the agent boots with `src/telemetry.ts` as first import, logs telemetry initialisation, and `/health` returns 200 (the Docker healthcheck) — i.e. the OTel SDK init did not throw.
- **Functional regression**: canonical O2C returns `lifecycleStatus: completed` / `reportState: fulfilled` (`LLM_PROVIDER=stub`).
- **Audit gate**: `npm audit --omit=dev` reports 0 vulnerabilities.
- **Note**: full end-to-end span *arrival* requires an OTLP collector/LangSmith backend; this ADR validates seam integrity (init + emit path) and functional non-regression, not backend ingestion.
