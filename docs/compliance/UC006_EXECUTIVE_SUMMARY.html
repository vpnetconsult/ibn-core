Intent-Based Networking with a Threat-Informed AI Gateway

How ibn-core turns a natural-language customer intent into a fulfilled network service under RFC 9315 — and how its inline AI-Gateway rejects unsafe requests before they ever reach the model or the network. End-to-end behaviour is captured as OpenTelemetry traces.

1 · Executive summary

One platform; one trace format; two opposite outcomes — fulfilment and principled refusal.

ibn-core is the open-core of an AI-native intent platform. A customer expresses an intent in plain language (“I need internet for working from home”); the platform translates it with an LLM, orchestrates fulfilment across back-office systems via the Model Context Protocol (MCP), and reports compliance — all mapped to the six phases of RFC 9315 §5 and the TMF921 Intent API.

Every request first passes an inline AI Gateway with three deny-capable controls. The same request path therefore produces four characteristic outcomes, each emitted as an OpenTelemetry trace with GenAI, RFC 9315-phase and AI-Gateway-event attributes:

2 · Architecture (C4 — container view)

System context: a Customer/API client drives ibn-core, which fans out to an LLM, three MCP back-ends and a Redis source-of-truth, while every component emits OTel spans to an OTLP backend.

3 · The intent-handling cycle & how context is managed

RFC 9315 §5 defines six phases. ibn-core models them as IntentHandlingCycleRunner phases over a single IntentHandlingContext.

Context discipline. IntentHandlingContext is immutable. Each phase receives the previous context and returns an enriched copy (object spread) — it is never mutated in place. Identity fields (intentId, customerId, intentText, startedAt) are readonly and carried unchanged end-to-end; each phase only adds the fields it is responsible for. This makes the state evolution explicit, the trajectory replayable, and a failure attributable to the exact phase that produced it.

4 · The AI Gateway — how the “reject” decision is made

Three independent, deny-capable controls. Each is rooted in a published control set and each emits a structured ai_gateway.* span event when it fires.

① Prompt-injection block

prompt-injection-detection.ts · OWASP LLM01

A signature engine matches the raw intent text against tiered regex patterns:

  • HIGH — jailbreaks: ignore previous instructions, pretend you are…, reveal your system prompt
  • MEDIUM — developer/admin mode, bypass safety, <script>
  • LOW — drop table, sudo, template/var injection

Rule: HIGH always blocks (blockHigh=true); MEDIUM/LOW are configurable. A block returns before translation — the model never sees the payload. → Scenario D.

② PII mask + DLP redact

pii-masking.ts + DLPPolicy.ts · MCP §4.4

Two layers protect data flowing to/from the model:

  • Input: the customer profile is PII-masked before any external AI call.
  • Output: DLP applies per-tool, per-role field allowlists to each MCP response; non-allowed fields become [DLP_REDACTED] (schema shape preserved).

Rule: e.g. on get_customer_profile a customer sees only non-identifying fields; raw email/phone are redacted. The cycle continues — this is containment, not refusal. → Scenario B.

③ Tool-policy RBAC deny

ToolPolicyEngine.ts · MCP §4.1 · RFC 9315 P5

Deny-by-default allowlists per role; the role is resolved from the caller’s API-key name:

  • customer → get_customer_profile, get_intent_status, get_quote, list_products
  • agent → + catalog, quote, submit_order, KG tools
  • admin → + cancel_order, update_customer_profile, manage_api_keys…

Rule: a tool not on the role’s allowlist is denied. A customer invoking the admin-only cancel_order is refused at orchestration. → Scenario C.

Two refusal semantics, deliberately different. A policy refusal (C) is a compliantFailure: the request was well-formed and understood, but the principal is not authorised — the system did the right thing by refusing, and records an auditable assessment (HTTP 403). A security block (D) is a hard reject at the boundary (HTTP 400): the input is hostile, so it is stopped before it can influence the model — there is no reportState because no intent was ever admitted.

5 · Inside the LLM step — what gets translated & what drives token cost

ibn-core makes two LLM calls (claude-client.ts), both single-turn JSON extractions at temperature/max_tokens fixed by config. Neither call ever sees raw PII — gate ② masks the profile first.

① analyzeIntent(intent, maskedProfile)

RFC 9315 §5.1.2 — TRANSLATING. The “understand” call.

In: a fixed extraction-instruction scaffold + the customer’s natural-language intent string + three masked profile fields (segment, current_services, tenure).

Out: structured JSON — tags (e.g. work_from_home), product_types (e.g. broadband), priority (e.g. speed, reliability). This is the literal “translation” of NL into network-actionable requirements.

② generateOffer({intent, products, bundles, segment})

RFC 9315 §5.1.3 — inside ORCHESTRATING, after the MCP fetches. The “decide” call.

In: a fixed offer-instruction scaffold + the intentAnalysis from ① + the BSS product catalog + the KG bundle list + the customer segment.

Out: JSON — selected_products, bundle_recommendation, pricing strategy, promotional discounts. This becomes the quote in the TMF921 response.

5.1 · What drives token usage

  • Input = fixed scaffold + interpolated runtime data, and the runtime data dominates. For ① it’s the intent length + the masked profile; for ② it’s the MCP catalog and bundle payloads — usually the single largest contributor. A bigger product catalog ⇒ more input tokens.
  • Output = the size of the returned JSON. More tags, more selected products, more discount lines ⇒ more output tokens.
  • Hard ceiling: max_tokens in ClaudeConfig caps output; temperature and model are fixed per deployment.
  • Masking shifts the count slightly: PII/DLP replace raw values with shorter sentinels before the prompt is built — e.g. Scenario B’s translate input is 298 tok vs A’s 312.
  • The gateway is the biggest lever of all. A call only happens if the request gets that far: D (blocked at ingest) = 0 LLM calls; C (denied before the offer) = 1; A/B = 2. Token spend scales with how far a request is allowed to travel — a refusal is also a cost saving.
